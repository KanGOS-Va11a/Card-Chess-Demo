# 开发规范（现行）

时间：2026-04-23

本规范从当前打磨阶段起生效，目标是解决此前反复出现的三类问题：

- 乱码和编码污染
- `msbuild` 不通过
- 场景 / 资源加载失败

同时，本规范也约束后续临时方案的使用，避免继续用会放大返工成本的短期修补方式推进项目。

## 1. 总原则

1. 所有新增实现必须服务于“完整可交付切片”，而不是继续扩散系统边界。
2. 所有代码修改必须优先保证工程稳定性，再考虑运行时表现。
3. 地图、战斗、成长、UI 的连接必须经过明确边界，不允许继续用临时直连覆盖主链。
4. 后续默认不由助手执行运行时内容审查；运行时表现由开发者本人在 Godot 内确认。

## 2. 编码规范

### 2.1 强制编码规则

- 所有文本文件统一使用 `UTF-8 without BOM`
- 所有文本文件统一使用 `LF`
- 禁止重新引入 `UTF-8 BOM` 到 `.tscn`、`.tres`、`.cs`、`.gd`、`.md`、`.cfg`、`.json`、`.csv`

### 2.2 新文件命名规则

- 新增代码文件、资源定义文件、导出配置文件优先使用 ASCII 命名
- 中文文件名只保留在已存在、且已稳定工作的美术源文件或策划文档中
- 新增运行时关键资源路径，禁止再使用乱码风险高的旧中文路径体系

### 2.3 文本写入规则

- 场景文件和资源文件不允许通过不明确编码的外部工具批量改写
- 修改中文文案时，优先直接修改 UTF-8 正常文件，不允许通过控制台复制乱码内容回写
- 如发现乱码，先定位编码链路，再修内容；禁止用“先替换成临时英文占位”长期悬挂

## 3. 构建规范

### 3.1 构建门槛

- 任何代码功能修改提交前，必须保证 `dotnet build` 通过
- 不允许带着已知 `msbuild` 错误继续叠加新功能
- 若某次需求跨多个系统，必须先做编译链路收口，再继续扩功能

### 3.2 警告处理策略

- 当前阶段不要求一次性清零全部历史 nullable warning
- 但新增修改不应继续显著扩大 warning 面
- 新功能引入的明显编译告警，应在同轮收口

## 4. 场景与资源稳定性规范

### 4.1 场景引用

- 运行时关键资源不得依赖目录扫描
- 运行时关键资源不得依赖 `FileExists(res://...)` 再决定是否加载
- 敌人贴图、音频、按钮贴图、关键 UI 场景等必须使用显式资源引用或稳定路径

### 4.2 场景修改

- 修改 `.tscn` / `.tres` 后，必须同步检查引用链是否仍成立
- 删除旧目录前，先确认当前主链没有引用
- 活跃主链场景以当前正式地图、正式战斗场景和正式 UI 为准；旧原型目录只可归档，不可混用

### 4.3 导出稳定性

- 所有历史目录、merge 目录、legacy 目录不得继续参与正式导出
- 导出预设必须使用合法 Windows 文件名
- 导出前优先排查：
  - 资源动态加载
  - 旧目录残留
  - 文本文件 BOM

## 5. 临时方案禁令

以下做法从现在起视为不合理临时方案，不应继续扩散：

1. 在格子制地图中继续使用球形碰撞作为核心交互判定
2. 对话、剧情、战斗入口分别走三套不同文本弹窗与锁输入逻辑
3. 用临时 debug 按钮、硬编码入口、运行时代码注入长期顶替正式系统
4. 返回地图时依赖整场景刷新来“顺带恢复状态”
5. 用局部 hack 修补构筑、奖励、学习、图鉴之间的数据同步问题，而不修正数据真源

## 6. 当前推荐实现方向

### 6.1 地图交互

- 地图交互底层应改为“以格子为准”
- 交互对象、剧情触发、出生点、敌人刷新点应尽量落到可绘制图层
- 判定逻辑优先基于格子、朝向、相邻关系，而不是球形碰撞包围

### 6.2 对话与剧情

- 所有剧情文本弹出方式统一
- 所有剧情文本弹出时，玩家输入必须被锁定
- 对话、剧情、战斗前说明、教程弹窗的触发边界必须一致

### 6.3 战斗返回地图

- 战斗返回地图不得刷新地图物品
- 地图物品、剧情触发区、敌人节点是否移除，应通过进度标记和快照控制
- 地图返回后的状态恢复必须基于显式 runtime snapshot，而不是重新实例化后的默认值

### 6.4 构筑系统

- 构筑数据必须有唯一真源
- 左侧候选区、右侧携带区、影响因子、战斗初始化必须全部读同一份结果
- 禁止继续使用“操作后靠其他行为触发刷新”的假同步逻辑

## 7. 运行时审查边界

- 后续默认不再由助手执行 Godot 运行测试
- 助手允许继续做：
  - 静态代码审查
  - 文档整理
  - 资源链检查
  - `dotnet build`
  - 场景/资源文本结构检查
- Godot 内的实际运行效果、UI 手感、触发位置、演出观感，由开发者本人审查确认

## 8. 变更记录规则

- 只有在“代码功能发生变化”时才写 `devlog`
- 纯文档新增、文档归档、目录整理、规范重写，不写 `devlog`
- 一次功能性修改对应一条 `devlog`
- 同一轮需求若只是继续收口同一功能，不重复拆出多条 `devlog`
