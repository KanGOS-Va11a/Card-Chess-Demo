# 地图格子化交互重构清单

时间：2026-04-23

## 1. 文档定位

本文件用于指导“地图交互底层从碰撞球近似判定迁移到格子化精确交互”的重构工作。  
它不是新的主任务文档，而是当前主任务中 `P0-地图交互改为以格子为准` 这一项的专项执行清单。

本文件只回答四个问题：

1. 最终结构应该长什么样  
2. 哪些现有方案继续保留  
3. 哪些现有方案必须废弃  
4. 重构应该按什么顺序推进

---

## 2. 目标结论

本项目不应继续把所有地图交互物都做成纯 `scene tileset`。  
推荐采用**混合结构**：

- 低状态、批量绘制、主要承担格子信息的对象：继续使用 `TileMapLayer`
- 高状态、强配置、带脚本流程的交互对象：改为**挂在地图 scene 下，但严格以格子坐标摆放和运行的节点**

换句话说：

- **保留 TileMap 的“格子 authoring 能力”**
- **放弃 TileSet scene instance 承担复杂状态逻辑**

---

## 3. 当前问题

## 3.1 玩家交互仍是“碰撞球 + 格子修正”

当前 [`Player.cs`](/D:/Godot/newproject/Card-Chess-Demo/Scripts/Map/Actors/Player.cs) 仍通过：

- `InteractionArea`
- `GetOverlappingAreas()`
- 再配合 `TryGetFacingTileTarget(...)`

做交互筛选。

这会带来：

- 交互边界不稳定
- 同一类对象摆放稍偏就会出现体验差异
- 地图设计时无法真正按格子精确思考
- 后续剧情区、敌人、箱子、NPC 会持续使用不同判定习惯

## 3.2 交互状态保存依赖节点路径

当前 [`MapRuntimeSnapshotHelper.cs`](/D:/Godot/newproject/Card-Chess-Demo/Scripts/Map/Transitions/MapRuntimeSnapshotHelper.cs) 通过**相对节点路径**保存交互状态。

这对手摆节点勉强可用，但对 TileMap 生成出来的 scene tile 实例不稳定，因为：

- 路径容易受层级和重构影响
- TileSet 实例名不适合作为长期状态主键
- 存档与地图改版之间更容易失配

## 3.3 `InteractionLayer` 当前承载了过多职责

当前 `InteractionLayer` 里混合了：

- 箱子
- 柜子
- 回复点
- 门
- 剧情触发

这些对象的共同特点是：

- 状态多
- 配置多
- 脚本差异大
- 后续还会继续扩展

它们已经不适合继续依赖 TileSet scene instance 作为长期 authoring 方案。

---

## 4. 最终目标结构

## 4.1 继续保留为 TileMapLayer 的内容

### A. `PlayerSpawnLayer`

保留原因：

- 只需要精确格子
- 几乎不带复杂状态
- 编辑器里直接绘制最方便

### B. `EnemySpawnLayer`

保留，但职责收缩为：

- 泛用敌人刷新锚点
- 遭遇入口
- 敌人类型标记
- 迷宫刷怪点

不要再让它承载：

- 特殊剧情敌人
- 教程敌人
- Boss 前置敌人
- 带独立流程的敌人交互对象

### C. 地图层纯格子信息层

继续保留为 TileMap 的内容包括：

- 出生点
- 泛用敌人刷新锚点
- 地图交互 footprint 预览
- 剧情 / 事件覆盖区域的可视化格子信息

说明：

- 这里讨论的只限**地图层**的数据。
- 战斗中的障碍、地形、缓行区属于 `BattleRoomTemplate` 和战斗房间 authoring 范畴，不属于本次地图格子化交互重构范围。
- 本次地图层重构不要顺手把战斗 TileSet 体系一起卷进来。

这些地图层信息本质上都是格子数据，继续 TileMap 化最合理。

## 4.2 必须改为格子化节点的内容

以下对象应全部迁移到地图 scene 下的节点容器中，不再作为纯 TileSet scene instance：

### A. 箱子 / 柜子 / 回复点 / 门

原因：

- 每个实例配置不同
- 需要 opened / empty / cooldown / granted items 等状态
- 后续还会继续接剧情和奖励

### B. NPC

原因：

- 每个 NPC 对话内容不同
- 可能有条件对话、一次性对话、商店、任务等扩展

### C. 剧情触发区

原因：

- 不是单纯碰撞体，而是高状态流程节点
- 需要绑定文本、战斗、切场景、一次性触发等行为

### D. 特殊敌人 / 教程敌人 / Boss 战入口敌人

原因：

- 它们通常不是“通用刷怪点”
- 而是带 encounter、剧情和状态语义的流程节点

## 5. 推荐的 authoring 方式

## 5.1 不推荐继续手摆自由节点

纯手摆自由节点的问题是：

- 位置不精确
- 后续对不齐格子
- 不方便统一调试

## 5.2 推荐做成“格子化节点”

每个高状态交互物都挂在地图 scene 下，但必须带统一格子字段，例如：

- `Cell : Vector2I`
- `InteractionId : string`
- 可选 `SizeInCells`

并统一具备：

- 编辑器下按格吸附
- 运行时按格定位
- 可视化显示占用格 / 触发格
- 用 `InteractionId` 保存状态

建议后续新增统一基类，例如：

- `GridMapObject2D`
- `GridInteractable2D`
- `GridTriggerZone2D`
- `GridEncounterAnchor2D`

注意：  
这里的关键不是类名，而是**所有高状态对象统一进入“格子坐标 + 稳定 ID + 节点配置”这套模式**。

---

## 6. 状态保存的目标改法

## 6.1 禁止继续以节点路径作为主状态键

后续地图交互状态不应再依赖：

- `sceneRoot.GetPathTo(interactable)`

作为长期主键。

## 6.2 新状态键规则

每个交互物必须有稳定键，优先级如下：

### 方案 A：显式 `InteractionId`

例如：

- `scene03_chest_01`
- `scene05_npc_dock_worker`
- `scene06_story_boss_intro`

这是首选方案。

### 方案 B：`ScenePath + Layer + Cell`

例如：

- `res://Scene/Maps/Scene03.tscn::interactable::12,7`

这是兜底方案。

## 6.3 需要保存的状态

至少统一支持：

- 是否已触发
- 是否已开启
- 当前交互次数
- 是否应移除
- 是否禁用
- 冷却状态
- 是否已清除 / 已击败

---

## 7. 对当前图层体系的调整建议

## 7.1 保留

- `PlayerSpawnLayer`
- `EnemySpawnLayer`

## 7.2 逐步废弃其“复杂交互职责”

### 当前 `InteractionLayer`

建议改成以下两种之一：

#### 方案 A：完全退出运行时交互主链

只作为历史层，后续不再继续使用。

#### 方案 B：改成“预览 / footprint 层”

只负责在编辑器里显示某类对象占格信息，不再承载真实状态对象。

对于当前项目，更推荐：

**中期保留为 footprint 层，长期退出主链。**

这样迁移成本最低。

## 7.3 新节点容器建议

在 `WorldObjects` 下统一增加：

- `GridInteractables`
- `GridTriggers`
- `GridSpecialEncounters`

其中：

- `GridInteractables`：箱子、柜子、回复点、门、NPC
- `GridTriggers`：剧情区、教程区、区域标题触发区
- `GridSpecialEncounters`：教程敌人、Boss 前置敌人、特殊脚本敌人

---

## 8. 分阶段重构步骤

## Phase 0：冻结旧交互扩散

目的：

- 在重构完成前，不再继续新增基于球形碰撞的交互对象

动作：

1. 停止新增依赖 `InteractionArea` 球形检测的新交互物
2. 停止把新剧情区继续做成 `Area2D` 自由矩形碰撞方案
3. 停止把新高状态对象继续塞进 `InteractionTileSet`

完成标准：

- 从这一阶段开始，所有新交互需求默认进入格子化重构链路

## Phase 1：建立格子交互统一基类

目的：

- 先把基础设施做出来，再迁移现有对象

动作：

1. 新增统一格子对象基类
2. 支持：
   - `Cell`
   - `InteractionId`
   - 编辑器吸附
   - 运行时定位
3. 为高亮、提示、状态快照预留统一接口

完成标准：

- 后续任意交互物都能基于同一套格子化基类实现

## Phase 2：玩家交互逻辑切换到纯格子判定

目的：

- 让玩家不再依赖重叠碰撞区找对象

动作：

1. 从 `Player.cs` 中移除“交互主要依赖 `Area2D` 重叠”的逻辑
2. 改为：
   - 读取玩家当前格
   - 读取面朝方向
   - 查询目标格 / 邻接格上的格子交互对象
3. 将提示文本、高亮、实际交互统一走格子查询结果

完成标准：

- 玩家交互逻辑与碰撞球脱钩

## Phase 3：先迁移四类核心对象

优先迁移：

1. `Chest`
2. `Npc`
3. `HealStation`
4. `StoryTriggerZone`

原因：

- 它们最能代表当前几种交互形态
- 迁完之后能覆盖：
  - 箱子式
  - NPC式
  - 进入范围触发式
  - 教程 / 剧情式

完成标准：

- 主流程已不依赖旧碰撞方案完成上述四类交互

## Phase 4：状态保存从路径键切到稳定 ID

动作：

1. 修改快照系统，不再以节点相对路径为主键
2. 改为：
   - `InteractionId`
   - 或 `ScenePath + Cell`
3. 同步更新：
   - 已打开箱子
   - 已触发剧情区
   - 已清除敌人
   - 已使用回复点

完成标准：

- 地图结构调整后，状态恢复仍稳定

## Phase 5：迁移敌人交互

动作：

1. 泛用迷宫敌人仍保留 `EnemySpawnLayer`
2. 特殊剧情敌人 / 教程敌人 / Boss 前置敌人迁移到格子化节点层
3. EncounterId、战斗前文本、移除逻辑统一到节点配置

完成标准：

- 敌人交互不再混用“图层锚点”和“自由节点战斗入口”两套逻辑

## Phase 6：决定 `InteractionLayer` 最终命运

最终二选一：

### A. 彻底退役

适合：

- 全部核心交互都已迁移完成

### B. 保留为编辑器预览层

适合：

- 仍希望在地图上可视化交互覆盖格

建议：

当前项目优先采用 **B**，等系统稳定后再考虑是否彻底移除。

补充限制：

- 这里的 footprint 仅指地图交互 footprint。
- 不包括战斗中的障碍、地形、缓行格或战斗房间题面信息。

---

## 9. 本任务的验收标准

只有满足以下条件，才能认为地图格子化交互重构阶段完成：

1. 玩家交互不再依赖球形碰撞重叠作为主链  
2. 箱子、NPC、回复点、剧情触发区已迁入格子化节点体系  
3. 状态保存不再以节点路径作为长期主键  
4. 主流程地图中可以按格精确摆放和调试交互物  
5. 前置 UI 弹出时，输入锁与交互锁已与新交互体系一致  
6. 战斗返回地图后，一次性交互与物品状态不会异常刷新

---

## 10. 当前阶段的明确建议

一句话建议：

**不要把高状态交互物继续做成 `scene tileset`。**  
**改成挂在地图 scene 下、但以格子坐标为准的节点模式。**  
**TileMap 只保留给地图层低状态、可批量绘制的格子数据。**

**战斗中的障碍、地形、缓行区继续留在战斗房间 authoring 体系中，不纳入这次地图交互重构。**

这是当前项目最稳、最便于调试、也最适合继续承载剧情和状态扩展的结构。
