# 阶段总结-地图格子交互与对话系统

时间：2026-04-23

## 1. 文档定位

本文档记录当前地图侧两条核心系统的阶段状态：

1. 地图格子交互系统
2. 地图对话/剧情系统

目标是明确：

- 现在已经稳定到什么程度
- 哪些问题已经解决
- 哪些问题仍然保留
- 后续应该如何继续推进

## 2. 当前阶段总判断

当前项目已经从“原型式临时方案拼接”推进到“主流程底层统一、可以继续稳定扩内容”的阶段。

更准确地说：

- 地图主交互已经进入格子系统主链
- 地图主流程文本入口已经进入统一对话服务层
- 文本完成后的关键后续动作已经开始协议化

这三点成立后，后续新增地图内容不再需要回退到旧的球形碰撞交互和分散回调逻辑。

## 3. 地图格子交互系统现状

## 3.1 已完成

### 1. 玩家主交互已切到格子判定

当前玩家交互目标不再依赖 `InteractionArea.GetOverlappingAreas()` 作为主入口，而是按：

- 玩家当前格子
- 面朝方向
- 目标格占用

来决定是否可交互。

### 2. 高状态地图交互物已迁出旧交互层

以下对象已经进入显式格子节点 authoring 链路：

- chest / cabinet
- heal station
- npc
- door
- 固定敌人 / 教程敌人
- story trigger zone

主流程地图中原来 `InteractionLayer` 承担的真实运行时交互物，已经基本迁完。

### 3. 场景状态键已不再依赖节点路径

当前交互物状态键优先依赖：

- `InteractionId`
- 或 `scene path + cell`

不再把节点层级路径当成长期稳定主键。

### 4. 编辑器内已经可以实时看到关键格子参数

当前在编辑器中：

- 交互物可实时显示 `Cell`
- 剧情触发区可实时显示：
  - `OriginCell`
  - `TriggerSizeCells`
  - 覆盖范围

这使地图 authoring 已经从“靠猜位置”进入“按格精确摆放”。

## 3.2 仍保留的问题

### 1. 整图鼠标悬停格子坐标还没有

现在能看到“对象自己的格子坐标”，但还不能做到：

- 鼠标指到地图任意格
- 即时显示当前格子坐标

这会影响大地图快速布点效率。

### 2. 部分预制件内部仍保留兼容性碰撞节点

虽然主交互链路不再依赖旧碰撞检测，但部分预制件内部仍保留：

- `Area2D`
- `InteractionArea`

它们目前更多是兼容壳，不再是主判定来源。

### 3. SystemFeatureLabController 仍是技术债点

当前运行时提示逻辑已经切到干净 partial 文件，但原控制器大文件内部仍保留历史实现，只是不再是主链入口。

结论：

- 这不影响当前主流程
- 但它仍然是后续可以继续清理的技术债点

## 4. 地图对话/剧情系统现状

## 4.1 已完成

### 1. 已建立统一地图对话服务层

当前已新增：

- `MapDialogueRequest`
- `MapDialogueResult`
- `MapDialogueCompletionStatus`
- `MapDialogueService`

服务层统一负责：

- 播放对话
- 检查是否已有对话正在播放
- 锁定/恢复玩家输入
- 返回完成状态

### 2. 地图主流程文本入口已接入服务层

以下主流程调用方已经不再自己实例化 `DialogueSequencePanel`：

- `StoryTriggerZone`
- `Npc`
- `StoryDialogueEnemy`
- `Scene01TutorialController`
- `Scene04To05CutsceneController`

也就是说，地图主流程中的：

- 剧情区文本
- NPC 文本
- 战斗前文本
- 场景教程文本
- 过场文本

都已经进入同一条运行时服务链路。

### 3. 已建立统一的“文本完成后后续动作协议”

当前服务层已经支持统一动作：

- `None`
- `TriggerInteractable`
- `StartBattle`
- `ChangeScene`

这套协议已经接入：

- `StoryTriggerZone`
- `StoryDialogueEnemy`
- `Scene04To05CutsceneController`

因此现在“文本结束后做什么”已经不再是每个脚本各写一套。

### 4. 文本阻塞判断已开始依赖统一服务状态

当前：

- `MapDialogueService` 已维护活跃阻塞对话状态
- `MapTextBlocker` 优先依赖服务层状态
- `PagedTutorialPopup` 也已经纳入地图阻塞判断

这意味着地图输入阻塞不再只靠“枚举具体某个面板是不是 visible”。

## 4.2 仍保留的问题

### 1. 仍然是“统一运行时”，不是“统一 authoring”

当前 Day 2 做到的是：

- 播放入口统一
- 后续动作统一
- 文本阻塞状态开始统一

但还没做到：

- 所有剧情文本资源化
- 所有文本从脚本里迁出

所以现在的统一是“运行时统一”，不是“内容 authoring 完全统一”。

### 2. DialogueSequencePanel 仍是底层 UI 本体

当前统一服务层只是把调用方收束到一条链路上。

底层播放控件仍然是现有的：

- `DialogueSequencePanel`

这没有问题，但意味着：

- 如果以后要换成 `dialogue_manager 3`
- 或换另一种表现

应该替换服务层内部，而不是重新改所有调用方。

### 3. 统一后续动作协议已经覆盖当前地图主流程文本入口

当前地图主流程文本入口都已经通过 `MapDialogueRequest` 显式声明后续动作数组：

- `StoryTriggerZone`
- `StoryDialogueEnemy`
- `Scene04To05CutsceneController`
- `Npc`
- `Scene01TutorialController`

其中：

- `StoryTriggerZone / StoryDialogueEnemy / Scene04To05CutsceneController`
  已经使用非空动作完成：
  - 触发交互
  - 开始战斗
  - 切场景
- `Npc / Scene01TutorialController`
  当前默认仍是空动作数组，用于保持现有流程表现不变

### 4. 教程弹窗仍是独立运行时类型

当前：

- 线性剧情/对话使用 `DialogueSequencePanel`
- 教程使用 `PagedTutorialPopup`

这是当前阶段可以接受的，但后续如果文本表现继续增多，仍需要继续约束不要出现第三、第四种文本窗。

## 5. 关于 dialogue manager 3 的结论

当前结论仍维持不变：

- 不建议现在直接把主流程强切到 `addons/dialogue_manager`

原因：

1. 当前主流程刚完成运行时链路统一
2. 如果立刻切插件，会把 Day 2 从“统一入口”变成“重做 authoring 基础设施”
3. 现在更重要的是把：
   - 输入锁
   - 后续动作
   - 主流程文本逻辑
   彻底稳定下来

推荐策略：

1. 主流程继续使用当前 `MapDialogueService + DialogueSequencePanel`
2. 后续如果需要：
   - 支线 NPC 试点
   - 大量剧情资源 authoring
   再小范围桥接 `dialogue_manager 3`

## 6. 当前阶段已经解决的核心风险

### 1. 地图交互不再严重依赖位置碰撞运气

现在对象摆放和触发逻辑已经主要由格子参数决定，编辑器 authoring 稳定性明显提高。

### 2. 文本结束后的行为不再分散

当前至少核心主流程已经不再是：

- 每个脚本自己加 panel
- 每个脚本自己接 completed
- 每个脚本自己决定怎么开战斗/切场景

### 3. 后续继续扩展时，不需要再回到临时方案

无论你接下来要做：

- 新 NPC
- 新剧情区
- 新教程敌人
- 新地图门

底层都已经有统一路径可以接入。

## 7. 下一步建议

按优先级，下一步建议如下：

### P0

继续收 Day 2：

1. 统一剩余特殊脚本的文本完成后回调行为
2. 检查是否还有主流程文本入口绕开服务层
3. 视需要为 `Npc / Scene01TutorialController` 补充非空 follow-up action authoring

### P1

开始做文本内容 authoring 升级：

1. 把脚本中的剧情文案逐步迁出
2. 为后续是否桥接 `dialogue_manager 3` 做最小试点

### P2

补编辑器 authoring 体验：

1. 整图鼠标悬停格子坐标显示
2. 更直观的触发区格子绘制/编辑辅助

## 8. 阶段结论

当前地图系统已经进入“可以继续做内容”的状态，但还没有完全到“内容 authoring 极其顺手”的状态。

换句话说：

- 底层已经基本不再拖后腿
- 但编辑器辅助和文本资源化 authoring 仍然有继续提升空间

这是一个适合继续做主流程内容、同时边做边收口工具层的阶段。
