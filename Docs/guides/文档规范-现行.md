# 文档规范（现行）

时间：2026-04-23

本规范用于限制后续文档数量、避免重复文档继续堆积，并明确“什么情况下才允许新增 md 文档”。

## 1. 当前阶段允许长期维护的文档类型

当前打磨阶段，长期活跃文档只保留以下几类：

1. 一份主任务文档  
2. 一份主设计 Wiki  
3. 一份提案草案  
4. 一份 AI 协作日志  
5. 少量仍有效的接口 / 规则 / 模板文档

除此之外，不再默认新增独立阶段文档。

## 2. 当前阶段的唯一主任务文档

当前阶段只允许存在一份主任务拆分文档：

- [打磨阶段开发主任务-2026-04-23.md](/D:/Godot/newproject/Card-Chess-Demo/Docs/design/打磨阶段开发主任务-2026-04-23.md)

后续开发需求默认都先收敛到这份文档中，而不是继续新建：

- `Day3...`
- `XX实现清单...`
- `XX优先级...`
- `XX补充计划...`

这类重复文档。

## 3. 新增 md 文档的准入条件

只有满足以下条件之一，才允许新增新的 Markdown 文档：

1. 需要形成长期维护的正式规范  
2. 需要形成正式对外提交材料  
3. 需求跨越 3 个以上系统，且单份主任务文档无法清晰承载  
4. 需要为团队成员提供可重复复用的模板或数据规范

若不满足以上条件，则：

- 更新现有活跃文档
- 或直接写入代码内注释
- 或只写 `devlog`

## 4. 明确禁止的新文档类型

从现在起，以下文档类型禁止继续新增：

1. 仅服务于一天冲刺的重复清单  
2. 与现有主任务文档高度重复的优先级文档  
3. 同一个系统拆成多份轻微改名的方案文档  
4. 只为记录“今天准备做什么”但没有长期维护价值的 md  
5. 纯归档、纯搬运、纯整理行为对应的说明文档

## 5. devlog 规则

- `devlog` 仅记录代码功能变化
- 不记录纯文档修改
- 不记录归档搬运
- 不记录 README、规范、提案材料调整
- 若功能修改同时包含少量文档同步，仍只写一条功能 `devlog`

## 6. 归档规则

- 旧阶段文档统一进入 `Docs/archive`
- 归档文档默认不再持续维护
- 归档文档中的内部链接允许失效，不以保持完全可跳转为目标
- 若归档文档仍有长期参考价值，应在 `Docs/README.md` 中保留索引说明，而不是重新激活原文档

## 7. 当前阶段的文档边界建议

### 保持活跃

- 设计 Wiki
- 提案草案
- AI 协作日志
- 主任务文档
- 接口 / 规则 / 模板类文档

### 统一归档

- 初赛冲刺期清单
- merge 过程文档
- 旧提案支撑分析文档
- 已被新规范替代的阶段说明
