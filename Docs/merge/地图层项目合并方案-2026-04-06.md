# 地图层项目合并方案

时间：2026-04-06

适用背景：
- 当前主工程：`D:\Godot\newproject\Card-Chess-Demo`
- 对方完整地图层工程：`D:\Godot\newproject\Card-Chess-Demo\Merge\Card-Chess-Demo`
- 由于双方从较早版本分线开发，存在大量同名脚本和场景，不适合直接做整工程覆盖式合并。

本文目标：
- 明确当前 merge 风险边界
- 确定哪些模块可以优先迁入
- 确定哪些模块必须手工合并
- 明确哪些全局与战斗层脚本禁止直接导入

## 1. 审查结论

### 1.1 当前总体判断

不建议：
- 直接对 `Merge/Card-Chess-Demo` 做整工程目录覆盖
- 直接做普通 git merge 后解决冲突

原因：
- 脚本和场景的同路径重叠度很高
- 其中大量文件不是“完全一致”，而是“同名但内容已经分叉”
- 如果直接合，很容易把当前战斗层和对方地图层同时弄坏

更合适的策略是：
- 以当前工程为主干
- 把对方工程当作“地图内容与演出模块 donor”
- 采用“分批迁入 + 核心文件手工合并 + 不碰 battle 真源”的方式推进

## 3. 风险分级

### 3.1 高风险：禁止直接覆盖

以下内容不允许直接覆盖当前工程：

1. `project.godot`
- 当前主工程的主场景与 battle 测试入口已经有自己的一套配置
- 对方项目的 `run/main_scene` 采用 UID 写法
- 直接覆盖会把当前入口、插件、显示配置一起改掉

2. `AutoLoad/GameSession.cs`
- 文件：
  - [当前 GameSession.cs](/D:/Godot/newproject/Card-Chess-Demo/AutoLoad/GameSession.cs)
  - [对方 GameSession.cs](/D:/Godot/newproject/Card-Chess-Demo/Merge/Card-Chess-Demo/AutoLoad/GameSession.cs)
- 两边文件哈希不同
- 虽然结构相近，但不能假设完全一致
- 必须保留当前主干版本，再选择性抄回对方新增字段或方法

3. `AutoLoad/GlobalBattleContext.cs`
- 文件：
  - [GlobalBattleContext.cs](/D:/Godot/newproject/Card-Chess-Demo/Merge/Card-Chess-Demo/AutoLoad/GlobalBattleContext.cs)
- 这是对方 battle/map 交接用的全局对象
- 和当前主干里的 [GlobalGameSession.cs](/D:/Godot/newproject/Card-Chess-Demo/Scripts/Battle/Shared/GlobalGameSession.cs) 概念重叠
- 不能直接并存使用，否则 battle 请求、返回场景、遭遇 ID 会出现双真源

4. 对方 `Scripts/Core`
- 文件：
  - [Core/BattleRequest.cs](/D:/Godot/newproject/Card-Chess-Demo/Merge/Card-Chess-Demo/Scripts/Core/BattleRequest.cs)
  - [Core/BattleResult.cs](/D:/Godot/newproject/Card-Chess-Demo/Merge/Card-Chess-Demo/Scripts/Core/BattleResult.cs)
- 当前主干已经有自己的 `Battle/Boundary` 层

## 5. 推荐 merge 路线

### 第一步：低风险迁入

先迁以下内容，不碰 battle 真源：

1. 对方独有地图场景
2. 对方独有地图视觉/教程/运行时快照脚本
3. 对方独有角色和交互 prefab，保留原目录结构

目标：
- 先让地图资源和演出能力进主工程
- 不破坏现有战斗层和全局状态层

### 第二步：手工合并地图核心脚本

建议顺序：

1. [MapSceneController.cs](/D:/Godot/newproject/Card-Chess-Demo/Scripts/Map/Controllers/MapSceneController.cs)
- 这是地图返回、战斗恢复、运行时快照回写的核心节点

2. [Player.cs](/D:/Godot/newproject/Card-Chess-Demo/Scripts/Map/Actors/Player.cs)
- 关系到复古 RPG 走格、输入、朝向、移动动画和剧情驱动

3. 交互基类与互动对象
- `IInteractable`
- `InteractableTemplate`
- `BattleEncounterEnemy`
- `SceneDoor`
- `Chest`
- `Npc`
- `HealStation`

目标：
- 以当前主干 battle 接口为准
- 把对方地图功能和演出逻辑吸收进来

### 第三步：地图接战斗

在第二步完成后，再统一做：
- 遇敌入口对接当前 battle request
- 战斗返回地图位置恢复
- 运行时地图状态快照回写

## 6. 明确约束

当前 merge 必须遵守：

1. 不改 battle 接口定义来迁就地图层
- 发生对接问题时，优先改地图层侧适配

2. 不引入第二套 battle 全局单例
- 不直接启用 `GlobalBattleContext`

3. 不直接替换 `project.godot`
- 只手工抄需要的设置

4. 不整目录覆盖 `Scripts/Map`
- 只能逐文件手工合并

## 7. 当前建议结论

当前最安全的 merge 方式不是普通合并，而是：

`以当前工程为主干 -> 先迁地图资源和独有脚本 -> 再逐个手工合 Map 核心脚本 -> 最后接回 battle`

这是当前最能控制风险、也最符合你“不要大量 merge 问题”的方案。
