# Demo 剩余代办与优先级清单 - 2026-04-12

## 0. 文档目的

这份文档不是泛泛的愿望列表，而是结合当前项目真实开发状态整理出的 demo 剩余任务清单。

判断标准不是“功能听起来重不重要”，而是：

- 是否直接阻塞 demo 主流程打通
- 是否已经有半成品，但还没有达到可展示状态
- 是否会在后续内容制作中持续放大返工成本

当前目标默认是做出一条完整、可展示、可录屏、可评审的 demo 主流程：

`主界面 -> 地图开场 -> 基础战斗教程 -> 逃跑教程 -> 荒川教程 -> 迷宫探索/遇敌 -> 存档补给 -> 中后段战斗 -> 最终 Boss -> 结尾演出`

## 1. 当前实际开发状态摘要

目前已经具备的基础：

- 地图与战斗的基础切换链路已经存在。
- `C` 键菜单、构筑、图鉴、天赋、装备、背包已经有可运行骨架。
- 基础战斗系统、敌人数据、房间池、卡牌运行时、地形、击退、战斗结算已经有较完整底座。
- donor 地图的大部分场景内容已经开始并入当前工程。
- 地图交互物的统一文字提示链路已经基本接回当前主工程。

目前最明显的风险：

- donor 地图合并仍未完全收尾，部分场景资源链、转场、排序、节点状态边界还在持续暴露问题。
- 教程流程虽然有若干底层能力，但还没有被正式组织成一个顺滑的引导链。
- 迷宫部分还没有真正的“敌人刷新与遇敌系统”，当前更接近静态摆放敌人。
- 存档、主界面、Boss、过场演出都还没形成可展示闭环。

结论：

- 现在项目已经不是“从 0 到 1”的阶段，而是“从能跑的开发版到可展示 demo”的阶段。
- 优先级必须围绕主流程闭环，而不是继续无边界扩系统。

## 3. P0 任务详列

### P0-1. donor 地图合并收尾与场景稳定化

这是所有后续地图内容制作的前置条件。

目标：

- 让当前确定采用的 donor 地图场景全部达到“可稳定进入、可稳定退出、可稳定交互”的状态。
- 地图中的门、交互物、灯光、排序、资源链、战斗入口不再持续冒新错误。

范围：

- `Scene01`
- `Scene03`
- `Scene04`
- `Scene05`
- `Scene06`
- `Scene07`
- `Scene08`

实际需要做的内容：

- 继续补 donor 地图剩余资源链，直到场景切换不再出现资源缺失。
- 收尾地图中 `Enemy`、`Door`、`Chest`、`Cabinet`、`Npc` 等交互物的状态恢复与移除逻辑。
- 完成 donor 地图中门传送目标、过场、背景、灯光、排序的统一校验。
- 统一地图交互提示形式，不再出现旧 `GalDialogueOverlay` 和新提示混用。
- 统一地图层的 `y sort` 规则，主角、交互物、装饰物不再持续出现层级错乱。
- 清理 donor 地图中的调试残留，例如碰撞可视、旧提示层、无效节点状态。

验收标准：

- 可以从 `Scene01` 一路通过门和战斗切到后续主线场景，不因缺资源或 TileMap 数据损坏中断。
- donor 主线地图中的主要交互物全部可正常交互。
- 地图中不再频繁出现“切到新场景立刻报一屏错误”的情况。

### P0-2. 基础战斗教程

目标：

- 完成最基础的一场战斗引导。
- 这场战斗不包含逃跑、不包含荒川能力。
- 在这场战斗中直接不显示荒川能量和荒川能力按钮。

实际需要做的内容：

- 单独做一份教程战斗 encounter / room / UI 配置，而不是用通用战斗全部功能硬塞。
- 给教程战斗增加阶段控制：
  - 只允许移动
  - 只允许普通攻击
  - 只允许打牌
  - 结束回合
- 教程战斗里隐藏：
  - 荒川能量显示
  - 荒川能力按钮
  - 逃跑入口
- 补一组教程提示文本：
  - 棋盘移动
  - 近战攻击
  - 出牌
  - 结束回合
- 教程失败 / 重试 / 胜利后回到地图的反馈也要做清楚。

验收标准：

- 玩家第一次进入战斗时，不会看到还没教的系统。
- 这场战斗能够自然引导玩家完成一次基础胜利。

### P0-3. 逃跑教程

目标：

- 单独教会玩家“移动到逃跑格并逃跑”的规则。

实际需要做的内容：

- 设计一个短教程战斗，只突出：
  - 识别逃跑格
  - 走到逃跑格
  - 使用逃跑动作
- 在这场战斗里明确告诉玩家：
  - 不是按按钮就能逃
  - 必须先踩到对应格子
- 教程文案与 HUD 提示要反映“逃跑格”和普通地块的差异。
- 逃跑后的结算界面也要纳入教程路径验证。

验收标准：

- 玩家能在不猜规则的情况下完成一次成功逃跑。

### P0-4. 荒川能力教程

目标：

- 单独教会玩家荒川能力是什么、按钮在哪、能对战场做什么。

实际需要做的内容：

- 单独做一场或一段分阶段教程，展示至少 1 个荒川控场能力和 1 个强化能力。
- 打开荒川系统时，明确说明：
  - 荒川能量是什么
  - 如何消耗
  - 对战场的影响
- 教程中要用到真实功能，而不是只弹说明文字。
- 需要一套教程专用 UI 显隐策略：
  - 基础战斗教程时隐藏
  - 到荒川教程时再解锁显示

验收标准：

- 玩家第一次接触荒川时，不会把它看成普通卡牌按钮的附属物。

### P0-5. 迷宫敌人刷新与遇敌

目标：

- 真正做出迷宫中的敌人刷新和遇敌，而不是只在地图上静态摆几个敌人。

当前判断：

- 现在 donor 地图更多是“静态摆放敌人 + 可交互进入战斗”。
- 没有完整的迷宫运行时刷怪系统。

实际需要做的内容：

- 定义迷宫中敌人刷新规则：
  - 固定点刷新
  - 房间池刷新
  - 是否按层/区块控制
- 让每个迷宫区域的敌人能绑定 EncounterId。
- 将地图敌人与战斗房间池对接彻底做实：
  - 地图敌人 -> EncounterId
  - EncounterId -> 房间池
  - 房间池 -> 房间随机抽取
- 明确“击败后消失 / 逃跑后是否保留 / 存档后如何恢复”的规则。
- 如果 demo 期不做真正 roaming AI，也至少要做稳定的静态刷新和唯一消耗逻辑。

验收标准：

- 迷宫区域的敌人能够按设计刷新，并稳定触发对应战斗。

### P0-6. 存档点兼回复点

目标：

- 地图中的存档点同时承担恢复与检查点功能。

实际需要做的内容：

- 做出正式的存档点交互：
  - 存档
  - 恢复生命
  - 恢复荒川能量
  - 更新检查点
- 失败 / 退出 / 重新进入时能回到正确检查点。
- 存档点在地图上的视觉与提示文案要明确。
- 迷宫中最好把存档点作为节奏节点，而不是纯技术设施。

验收标准：

- 玩家明确知道哪里能保存与补给。
- 存档点可支撑 demo 从中段继续测试，不必每次从头跑。

### P0-7. 装备获取与地图交互获得物资整理

目标：

- 让地图探索获得物品、装备、材料的路径清晰可控。

实际需要做的内容：

- 梳理箱子、柜子、地面拾取、事件奖励这几类来源。
- 明确哪些是：
  - 装备
  - 消耗物
  - 材料
  - 关键物品
- 保证获得后能立即在背包/装备栏中看到。
- 对现有 donor 地图里的箱子和柜子内容做一次总表整理，避免重复和脏数据。

验收标准：

- 玩家在地图里拿到东西后，菜单和战斗都能正确反映。

### P0-8. 剧情提示与流程引导整理

目标：

- 让玩家知道“现在该做什么”，而不是系统都在但流程断裂。

实际需要做的内容：

- 统一剧情文本、教学提示、交互提示、章节目标提示的表现方式。
- 为每个主流程节点补充引导：
  - 去哪里
  - 为什么去
  - 到那里后要做什么
- 把 donor 地图中的剧情触发、门跳转、教程提示串成连续路径。
- 需要时补充地图上的短目标提示 UI。

验收标准：

- 主流程中不出现“玩家不知道下一步去哪”的断点。

### P0-9. 主界面

目标：

- 做出 demo 可展示的正式入口。

实际需要做的内容：

- 主界面至少包含：
  - 开始游戏
  - 继续游戏
  - 设置
  - 退出
- 接上当前已有音量设置。
- 进入游戏时接正确的开场地图或开场流程。
- 如果 demo 允许，补一个“从教程开始 / 从中段测试开始”的开发入口，但要和正式入口区分清楚。

验收标准：

- 玩家进入游戏不是直接落在开发场景，而是有正式入口。

## 9. 收工判断标准

只有满足下面条件，这个 demo 才算进入“可展示”状态：

- 玩家能从主界面进入游戏
- 玩家能完成三段式教程
- 玩家能进入迷宫并发生稳定遇敌
- 玩家能通过存档点继续推进
- 玩家能通过探索拿到装备/物资并在菜单中确认
- 玩家能打到最终 Boss
- 玩家能在 Boss 战后看到结尾反馈
- 全流程不再因为地图切换、资源缺失、状态恢复而中断

如果这些条件还没同时成立，就不应该过早沉进纯 polish 或额外扩系统。
