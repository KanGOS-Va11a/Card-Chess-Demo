# 快照语义与地图衔接补充

本补充专门说明这轮 `GlobalGameSession` 分层与“装备后攻击重复结算”修复，对 battle / map 衔接实际意味着什么。

## 1. 本次修复的真实问题

问题根因不是 battle 攻击动作本身打了两次，而是：

1. `BuildPlayerSnapshot()` 输出的是 battle 可直接使用的解析后攻击值
2. `BattleRequest.ApplyToSession()` / `BattleResult.ApplyToSession()` 又把这些解析后数值回写进了 `GlobalGameSession`
3. `PlayerStatResolver` 再基于装备和天赋继续结算一次

于是就会出现：

- 武器加成被重复算进玩家攻击
- 伤害表现为对护甲和生命重复放大

## 2. 当前修复后的快照语义

`BuildPlayerSnapshot()` 现在同时输出两组信息：

### 2.1 基础字段

- `base_max_hp`
- `base_move_points_per_turn`
- `base_attack_damage`
- `base_defense_damage_reduction_percent`
- `base_defense_shield_gain`

这组字段的用途是：

- 供 `ApplyPlayerSnapshot()` 在 battle request / battle result / save 恢复时写回 session 基础状态

### 2.2 解析字段

- `max_hp`
- `move_points_per_turn`
- `attack_damage`

这组字段的用途是：

- 供 battle 运行时直接读取可战斗数值

当前约束已经明确为：

- battle 可以继续依赖解析字段
- session 回写时优先依赖基础字段
- 不能再把解析后的攻击或生命直接当成基础值写回

## 3. 对地图层衔接的实际影响

### 3.1 地图进入战斗链路不需要改接口

地图仍然只需要：

1. 准备 `EncounterId`
2. 构建或触发 `BattleRequest`
3. 交给 `GlobalGameSession`
4. 切换 battle 场景

也就是说：

- 地图层不需要知道装备加成如何计算
- 地图层不需要知道 `PlayerStatResolver`
- 地图层也不需要自己维护一份解析后角色数值

### 3.2 battle 返回地图链路也不需要新增 DTO

战斗结束后：

- `BattleResult.PlayerSnapshot` 继续作为返回快照
- session 回写时已经会优先吃基础字段

因此地图层不用为了这次修复新增别的返回结构。

## 4. 为什么这次改动对地图层是“低侵入”的

因为这次真正变化的是：

- session 内部的状态恢复语义
- battle 内部的数值来源

而不是：

- 地图进战斗接口名
- 地图出战斗接口名
- `BattleRequest` / `BattleResult` 顶层结构

所以地图层需要注意的只有一点：

- 不要自己把 `attack_damage`、`max_hp` 这类 battle 字段再拿去当角色基础成长字段缓存

## 5. merge 时最需要看的文件

如果后续和地图层 merge，优先检查：

1. `Scripts/Battle/Shared/GlobalGameSession.cs`
2. `Scripts/Battle/Boundary/BattleRequest.cs`
3. `Scripts/Battle/Boundary/BattleResult.cs`
4. `Scripts/Battle/State/BattleObjectStateManager.cs`

重点确认：

- `BuildPlayerSnapshot()` 还同时带有 base 字段和解析字段
- `ApplyPlayerSnapshot()` 仍优先使用 base 字段
- state manager 仍通过 `ResolvePlayerStats()` 取玩家数值

## 6. 当前结论

这次修复后的正确理解是：

- 地图与战斗的边界接口没有被重构
- battle 内部状态恢复语义被修正了
- 装备 / 天赋导致的玩家属性修正，当前仍由 session + resolver 收口
- 地图层继续只依赖稳定边界，不需要同步跟着重写
