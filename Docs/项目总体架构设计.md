# 项目总体架构设计

## 1. 文档范围

本文件描述当前仓库里已经落地的真实结构，不把计划中的系统写成“已完成”。

当前工程由两块原型组成：

- 地图交互原型
- 独立战斗原型

两者已经通过 `GlobalGameSession`、`BattleRequest`、`BattleResult`、`MapResumeContext` 接成最小边界，现状是：

- 地图可发起战斗
- 战斗失败后可返回原地图位置
- 完整胜利结算、掉落、存档等仍未落地

## 2. 当前入口

- `project.godot` 当前主场景：`Scene/Mainlevel.tscn`
- 全局 Autoload：`GlobalGameSession`

这意味着默认启动工程时会先进入地图原型，而不是直接进入战斗测试场景。

- 保存玩家跨场景最小运行时状态

## 4. 当前地图 / 战斗边界

当前边界分工如下：

- 地图侧通过 `MapBattleTransitionHelper` 发起战斗切换
- 切换前写入 `BattleRequest`、`MapResumeContext`、`PendingBattleEncounterId`
- 战斗侧由 `BattleSceneController` 消费这些边界数据
- 战斗失败时写回 `BattleResult`
- 返回地图后由 `MapSceneController` 恢复玩家位置并消费结果

这是一个最小可运行闭环，不代表完整的总循环已经成型。

## 5. 当前战斗启动链路

1. 地图交互对象调用 `MapBattleTransitionHelper.TryEnterBattle(...)`
2. `GlobalGameSession` 记录待处理的战斗请求和地图恢复信息
3. 场景切换到 `Scene/Battle/Battle.tscn`
4. `BattleSceneController` 在 `_Ready()` 中消费边界数据并初始化战斗对象
5. 房间模板生成 `RoomLayoutDefinition`
6. `BoardInitializer` 初始化棋盘与对象注册表
7. `BattleObjectStateManager`、`BattlePieceViewManager`、`BattleHudController` 接上运行时状态和表现层

## 6. 当前已实现的核心能力

- 地图玩家移动与交互扇形判定
- 统一地图交互基类与冷却 / 禁用逻辑
- 地图敌人 / 门进入战斗
- 战斗失败返回地图
- 棋盘初始化、对象注册、占位规则
- 最小回合状态树
- 普通攻击、防御、冥想原型动作
- 最小卡牌循环与荒川能力
- 悬停信息、HUD、战斗表现层

## 7. 当前仍未完成的系统

- 完整胜利结算与掉落
- 存档 / 读档
- 更丰富的地图进度、事件与状态持久化
- 数据驱动牌库资源与正式构筑系统
- 更复杂的敌人 AI 和技能编排
- 更完整的地图 <-> 战斗双向结果闭环

## 8. 当前维护结论

后续继续迭代时，应遵守以下约定：

- 地图域代码继续放在 `Scripts/Map`
- 战斗域代码继续放在 `Scripts/Battle`
- 地图恢复逻辑只放在 `MapSceneController`
- 进入战斗的公共边界只放在 `MapBattleTransitionHelper` 与 `GlobalGameSession`
- 任何新增变更都要同步更新 `Docs` 与 `devlog`
