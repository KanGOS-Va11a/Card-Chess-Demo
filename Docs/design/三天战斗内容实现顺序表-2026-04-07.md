# 三天战斗内容实现顺序表

时间：2026-04-07

适用目标：
- 在 3 天内把当前 demo 的战斗部分推进到“可稳定展示”的状态
- 优先完成真正影响可玩性的内容，不把时间花在低回报系统深化上
- 默认前提：地图进入战斗、战斗返回地图、`C` 菜单基础入口已经可用

关联文档：
- [战斗设计实现优先级清单.md](D:/Godot/newproject/Card-Chess-Demo/Docs/design/战斗设计实现优先级清单.md)

## 一句话结论

这 3 天不要并行大做“战斗内容实现、战后结算深化、音频系统、存档系统”四条线。

正确顺序是：

1. 先把卡牌 / 敌人 / 装备 / 天赋影响做成真正可玩的战斗内容
2. 再把战后结算与奖励闭环接实
3. 最后补最小音频系统

存档系统这 3 天只保留现有最小骨架，不做深化。

## 总体原则

- 每天都必须有“当天可试玩结果”，不能把三天都耗在底层整理上
- 每天结束前都要能完整跑一遍 `地图 -> 战斗 -> 结算 -> 返回地图`
- 优先做设计文档中“改变题面”的内容，而不是单纯补数值
- 不在这 3 天内追求整棵天赋树、完整敌人池、完整存档版本管理

## Day 1：把核心战斗内容做成可玩闭环

### 当天目标

- 把设计文档里最关键的一批卡牌、敌人行为、装备影响补进现有战斗框架
- 明确主角普通攻击、武器、天赋对战斗数值的实际影响
- 做出一套能稳定测试的内容包，而不是零散样例

### 必做内容

1. 卡牌第一批落地
- 目标：每个分支至少有 2 张真正改变题面的牌可用
- 推荐优先：
  - 毁灭：`冲撞`、`重击`
  - 制衡：`快枪`、`震荡射击`
  - 创造：`电弧泄露`、`风化`
- `学习` 若当天来得及，先只保留接口与记录入口，不要求当日完全闭环

2. 普通攻击与武器边界做实
- 普通攻击必须受当前武器影响
- 天赋要能影响：
  - 攻击力
  - 防御减伤比例
  - 移动力或基础战斗参数中的至少一项
- `拔枪` 要和武器临时覆盖链路保持一致

3. 敌人第一批差异化
- 至少保证 3 类普通敌人有明确分工：
  - 贴身压迫
  - 远程直线威胁
  - 利用障碍 / 地形的单位
- 避免只通过血量和攻击力做假差异

4. 装备第一批落地
- 每个槽位至少 1 件真正生效的装备：
  - 武器
  - 护甲
  - 饰品
- 装备效果必须真实参与战斗，不只是显示在菜单里

### 当天交付物

- 一套可直接进入战斗测试的 debug 卡组
- 一间可稳定复现的测试房间
- 3 类普通敌人 + 3 件基础装备
- 普通攻击、卡牌、天赋、装备之间的最小影响链路

### 当天验收标准

- 玩家用不同装备进入 battle，普通攻击表现不同
- 同一套房间中至少能打出两种明显不同的打法
- 战斗中不会因为新增卡牌或装备导致入口链路断掉

### 风险控制

- 如果卡牌数量做不完，优先保证“每个流派 2 张代表牌”，不要追总量
- 如果天赋节点 UI 还不稳定，先保证天赋数值生效，再优化展示

## Day 2：把战后结算与成长闭环接实

### 当天目标

- 把“战斗打完之后玩家到底得到什么”做成稳定结果
- 把 `学习` 机制、经验值、基础奖励、失败回退逻辑接到现有 session 上

### 必做内容

1. `学习` 机制最小闭环
- 记录敌人上回合实际使用动作
- `学习` 卡读取该记录
- 胜利后根据记录结果解锁：
  - 普通学习牌
  - 或特色学习牌

2. 战后奖励统一入口
- 经验值
- 解锁卡牌
- 装备 / 物品占位奖励
- 运行时标记，例如剧情标志、首通标志

3. 战败 / 逃跑 / 胜利三种结果校正
- 胜利：应用奖励并返回地图
- 战败：按当前回退策略回到指定存档点或检查点
- 逃跑：仅带出允许保留的状态，不错误发放胜利奖励

4. 地图与菜单状态回写
- 结算后 `GlobalGameSession` 中的以下内容要正确更新：
  - 经验
  - 已学习卡牌
  - 当前牌组
  - 装备状态
  - 战斗结果标记

### 当天交付物

- 1 个支持学习的精英敌人
- 1 套稳定的战后奖励结果
- 胜利 / 战败 / 逃跑三分支都可验证

### 当天验收标准

- 从地图进战斗，结算后返回地图时，菜单能看到状态变化
- 学习成功时，卡牌图鉴或构筑列表中能看到新增内容
- 战败不会错误保留本场不应保留的收益

### 风险控制

- 如果奖励池内容不够，先用固定奖励表，不要把时间花在掉落随机化
- 如果学习特例太多，先只接 1 个精英和 1 个 Boss 的学习奖励

## Day 3：补最小音频系统并做展示级整体验收

### 当天目标

- 给战斗 demo 补上最小可展示音频系统
- 把前三天内容串成一条完整展示链，并修掉明显破坏体验的问题

### 必做内容

1. 最小音频系统
- 至少区分：
  - 地图 BGM
  - 战斗 BGM
  - 按钮 / 交互音效
  - 攻击 / 受击 / 击杀音效
  - 战斗开始与结束转场音效
- 先不追复杂混音和自适应分层音乐

2. 关键触发点接线
- 进入战斗时切 BGM
- 战斗结束返回地图时恢复
- 关键动作触发对应 SFX：
  - 普通攻击
  - 卡牌攻击
  - 地形伤害
  - 击杀
  - 学习成功

3. 最终通路验证
- `地图探索 -> 遇敌 -> 战斗 -> 结算 -> 返回地图 -> C 菜单检查成长`
- 连续跑多次，排查 session 污染、奖励重复结算、菜单不同步

4. 修最后一批高影响 bug
- 优先级只看是否破坏展示：
  - 战斗进不去
  - 卡牌打不出
  - 结算不回写
  - 音乐切换卡死
  - 菜单数据显示错误

### 当天交付物

- 一条完整可录屏的 demo 通路
- 最小战斗音频系统
- 一份最终问题清单，区分：
  - 比赛前必须修
  - 比赛前可接受

### 当天验收标准

- 从主场景起步，至少连续完整跑 3 次不卡死
- 战斗开始、关键动作、战斗结束都有音频反馈
- 菜单里看到的成长与装备状态和战斗结果一致

### 风险控制

- 如果音频素材不齐，先把总线、播放器、触发点和占位资源接好
- 如果 Day 2 仍有高风险 bug，Day 3 先补稳定性，再补音频表现

## 三天内明确不做的事

- 深化存档系统版本管理
- 完整手动存档 / 自动存档 UI
- 大规模扩房间池
- 整棵天赋树全部节点数值化
- 大规模敌人图鉴与卡牌图鉴内容填充
- 大规模美术 polish

原因不是这些不重要，而是它们对“3 天内做出能打、能结算、能展示的 demo”回报太低。

## 推荐每日收工检查表

### Day 1 收工前
- 能进 battle
- 新卡牌能打
- 装备生效
- 普通敌人差异成立

### Day 2 收工前
- 胜利 / 战败 / 逃跑都能正确结算
- 学习机制至少在 1 个精英上跑通
- 返回地图后菜单数据一致

### Day 3 收工前
- 地图与战斗音频切换正常
- 完整流程可录屏演示
- 高影响 bug 清单明确

## 最后建议

如果这 3 天你只能守住一件事，那就守住这条链：

`遇敌 -> 小棋盘策略战斗 -> 学习 / 奖励 / 成长 -> 返回地图继续构筑`

只要这条链稳定成立，哪怕音频和存档还只是最小版，这个 demo 也已经具备展示价值。
