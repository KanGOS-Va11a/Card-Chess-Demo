# AI协作日志

## 1. 协作定位

本项目的核心创意、题材判断、玩法取舍、世界观方向与最终体验目标均由开发者主导。  
AI在本项目中承担的角色，不是“替代作者完成创作”，而是一个高频参与、可快速响应的技术与设计协作伙伴。

由于本作同时包含：

- 卡牌构筑
- 小棋盘战斗
- 敌人学习机制
- 装备与天赋成长
- 地图探索与剧情触发
- 多场景切换与存档回写

这些系统之间存在较强耦合，传统独立开发中往往会在“想法明确但实现成本过高”处出现推进停滞。  
AI协作的价值，主要体现在帮助开发者把大量“知道要做什么，但一个人很难快速搭出第一版”的系统，迅速推到可验证状态。

换句话说，AI在本项目中最重要的作用不是替我“想一个游戏”，而是帮助我把已经清晰的设计，尽快落成引擎内真实可玩的版本。

---

## 2. AI参与的主要工作范围

结合当前仓库中的 `Docs`、`devlog` 与持续开发过程，AI主要参与了以下几类工作：

### 2.1 程序实现与重构

- 协助在 Godot 4 + C# 环境下实现战斗、地图、菜单、成长、音频等系统
- 在系统不断扩展时，帮助做边界收束与模块拆分
- 将容易耦合在单个控制器中的逻辑，逐步整理为：
  - 请求/结果结构
  - 运行时状态对象
  - 配置资源
  - 地图层与战斗层之间的标准接口

这一点可以从项目中大量接口文档、状态快照说明和服务层拆分中看出。  
例如：

- `GlobalGameSession`
- `BattleRequest / BattleResult`
- `MapResumeContext`
- `BattleResolutionService`
- `SaveGameService`

这些都不是单纯“把功能写出来”，而是为了让后续地图、成长、奖励、存档继续扩展时仍然能接得住。

### 2.2 设计文档整理与落地矩阵

AI不仅参与代码，还大量参与了设计文档的结构化整理。

项目中已有的：

- Day1/Day2冲刺清单
- 设计Wiki全量落地矩阵
- 敌人数值与行为设计文档
- 卡牌、天赋、装备模板
- 当前项目进度总览
- 项目接口文档
- 地图层合并方案与执行记录

这些文档的价值在于：

- 把“想法”转成“可执行条目”
- 把“功能名称”转成“资源、脚本、接口、验收条件”
- 把“今天做什么”与“之后如何扩展”区分开来

对于独立开发来说，这种文档整理本身就是生产力工具。AI在这里承担了“把思路整理成可执行开发语言”的角色。

### 2.3 调试、排错与版本收束

本项目后期大量问题并不是“不会写功能”，而是：

- merge后资源链断裂
- 场景挂载失效
- 旧逻辑回流
- 编码污染
- 导出阶段资源丢失
- 动态加载路径在打包后失效

AI协作的重要作用，是帮助快速定位问题来源，并给出更适合当前项目状态的修复路线，例如：

- 判断问题是逻辑问题、资源问题还是导出问题
- 区分“当前生效链路”和“历史残留目录”
- 用文档、日志和代码同时交叉确认真实依赖
- 在修复过程中尽量避免误伤现有主流程

这让项目在高频试错中仍然能保持推进，而不是反复陷入回滚和重做。

### 2.4 音频系统与交互反馈接入

由于开发者本身对游戏音频较为重视，AI在音频相关工作中承担了较多“技术实现与挂载”职责，例如：

- 基础 BGM / SFX 播放系统搭建
- 场景与战斗音乐切换
- 按钮选中、取消、出牌、受击、死亡等音效绑定
- 淡入淡出
- 总线效果器与受击闷声效果
- 设置界面中的分轨音量与静音逻辑

这类工作本身并不决定项目题材和审美，但对最终成品感影响极大。AI帮助把音频表现从“有资源但未接入”快速推进到“系统级可用”。

## 3. AI协作中最有价值的一点：快速开发引擎内设计工具

这是本项目里最重要、也最能体现AI协作价值的部分。

我并没有把AI主要用于“替我生成大量内容”，而是大量用于**快速搭建引擎内可编辑、可验证、可复用的设计工具**。  
这些工具直接改变了后续内容制作方式，让项目从“硬编码原型”转向“能在Godot编辑器里持续产出内容”的状态。

### 3.1 战斗房间作者工具

在战斗系统早期，如果每个战场都靠代码写死，就会导致：

- 调整效率极低
- 房间差异难以验证
- 后续敌人扩展非常痛苦

AI协作帮助我快速把这套流程改造成：

`TileMap房间预制 -> 场景保存 -> 房间池资源 -> 按敌人类型随机抽取`

这条作者链路。

具体落地包括：

- `BattleRoomTemplate`
- `FloorLayer`
- `MarkerLayer`
- `FacingLayer`
- `EscapeLayer`
- `BattleRoomPoolDefinition`
- `BattleRoomPoolEntry`
- `BattleRoomTileSetFactory`

这意味着后续设计战斗不再是改代码，而是可以在引擎内直接绘制：

- 玩家/敌人出生点
- 三类障碍物
- 地形分布
- 单位朝向
- 可逃跑区域
- 不同敌人的专属房间池

对于策略游戏来说，这一步极其关键，因为它把“系统可玩”推进成了“内容可持续生产”。

### 3.2 地图敌人生成与遭遇工具

在地图探索部分，AI协作帮助我把“场景里摆一个敌人然后进固定战斗”逐步升级为：

- 地图敌人场景
- EncounterId
- 房间池映射
- 敌人刷新锚点
- 迷宫生成控制器

这一整套工具化流程。

后续又进一步发展为：

- `EnemySpawnLayer`
- 敌人 TileSet
- 可配置多种敌人的刷新锚点
- 地图敌人进入战斗后，根据 EncounterId 进入对应房间池

这一改动的意义，不只是“方便摆敌人”，而是把地图设计与战斗内容设计真正连接起来。  
从此以后，关卡设计者在地图里放的不是一坨临时节点，而是能直接驱动战斗配置的内容入口。

### 3.3 交互层与剧情触发工具

类似地，地图中的交互装置也被逐步工具化，而不是继续散落在各处手工挂节点。

AI协作帮助搭出的作者工具包括：

- `InteractionLayer`
- 交互 TileSet
- 箱子/柜子/治疗站统一交互链路
- 一次性交互文本与奖励数组
- `StoryTriggerZone`
- 玩家出生点图层 / 出生标记

这些工具使得剧情和地图交互从“写在某个临时脚本里”变成：

- 可在编辑器中放置
- 可配置触发范围
- 可配置文本页
- 可配置后续行为（进入战斗/切场景/触发目标）

这对竞赛版尤其重要，因为它让教程、空港、迷宫、Boss前过场都能在统一工具链中生产，而不是每段都重写一套逻辑。

### 3.4 成长与局外内容的可视化工具

AI协作还帮助把局外系统从“调试用原型”往“正式可展示界面”推进，包括：

- C键综合菜单
- 图鉴页签与解锁逻辑
- 卡组构筑界面
- 天赋树面板
- 角色状态与分系能力显示

其中天赋树和构筑界面的价值，不只在于“玩家能看”，还在于它们本身已经成为**设计验证工具**：

- 我可以立即看到某个天赋是否真实生效
- 我可以测试某张卡加入/移除卡组后的构筑反馈
- 我可以验证成长、解锁、图鉴显示是否一致

也就是说，AI帮助搭出的这些UI，并不只是最终展示面板，同时也是持续开发中的“设计验证面板”。

## 4. AI协作的实际工作流

本项目中，AI并不是独立工作，而是嵌入在日常开发工作流里的。

一个典型流程通常是：

1. 我先给出非常具体的目标或限制条件  
   例如：
   - 某个界面要怎么改
   - 某个敌人要怎么行动
   - 某个地图工具要如何在编辑器里使用
   - 某个系统应该通过哪条链路接入，而不是另一条链路

2. AI先阅读当前代码与文档  
   不凭空假设，而是：
   - 看脚本
   - 看场景
   - 看资源
   - 看现有接口文档
   - 看 devlog 中的上下文

3. AI把需求转成结构化修改  
   例如：
   - 新增资源定义
   - 补脚本接口
   - 调整场景层级
   - 修改作者工具的图层与 TileSet
   - 修复 battle/map 边界

4. 我在引擎里直接验证  
   核查：
   - 手感是否对
   - 布局是否顺眼
   - 流程是否符合设计预期
   - 是否需要继续收口

5. 再由AI继续跟进修正  
   包括：
   - 调整数值
   - 收缩路径
   - 修资源引用
   - 处理导出与编码问题

这种工作流最大的特点，是可以支持非常高频、非常具体的迭代。  
尤其适合独立游戏开发里“需求变化快、验证粒度细”的阶段。

## 5. AI协作带来的实际收益

### 5.1 把“想法验证时间”压缩到很短

如果没有AI，很多系统虽然能想清楚，但真正把第一版做出来要花大量切换成本。  
而在本项目中，AI使很多功能能够快速从“口头目标”进入“引擎内可跑版本”。

尤其是：

- 房间作者工具
- 地图图层工具
- 遭遇与房间池映射
- 剧情触发工具
- 图鉴/构筑/天赋菜单
- 战斗结果与存档链路

这些都属于“不做就无法扩内容，但手工写又很耗时”的基础设施。

### 5.2 提高单人/小团队项目的系统承载能力

本作并不是单一机制游戏，而是多个系统并行推进。  
AI协作的意义在于，它让我能同时维护：

- 战斗系统
- 地图系统
- 成长系统
- UI系统
- 音频系统
- 文档系统

而不至于因为系统过多而完全失控。

### 5.3 让文档、代码、设计三者保持同步

项目中始终保留了 `Docs + devlog` 双轨记录。  
AI在这里帮助我持续做两件事：

- 把“今天做了什么”写成可追踪日志
- 把“系统现在真实是什么状态”沉淀为正式文档

这不仅提升我自己的开发清晰度，也让后续与队友协作、与外部评审说明时，有真实材料可依。

## 6. AI协作的边界与原则

虽然AI在本项目中占据了较高比重，但有几条边界始终没有改变：

### 6.1 创意主导权仍由开发者掌握

以下内容始终由开发者主导：

- 题材与世界观
- 核心玩法方向
- 产品定位
- 节奏判断
- 视觉风格取舍
- 最终版本是否采用某个方案

AI更适合参与的是“如何快速实现”和“如何把结构理清”，而不是替代作者做最终设计判断。

### 6.2 AI生成的内容必须经过人工验收

不论是：

- 系统逻辑
- UI改动
- 地图链路
- 文本文案
- 导出修复

都必须在引擎内和项目链路中实际验证。  
本项目里大量迭代，都是“AI先给出实现 -> 人工试跑 -> 再继续修正”的循环，而不是一次生成后直接采用。

### 6.3 AI最适合做“加速器”，不是“方向盘”

对于独立游戏来说，最危险的不是AI帮得太多，而是开发者把方向判断也完全交出去。  
本项目中的经验恰恰相反：

- 方向必须人来定
- 玩法品味必须人来兜底
- AI最有价值的时候，是被用来把方向快速推成可玩的东西

## 7. 对本项目最关键的AI协作总结

如果只用一句话总结本项目中AI的最大价值，那就是：

**我并不是主要用AI去“替我做内容”，而是用AI快速搭建可持续生产内容的引擎内设计工具。**

这使项目的推进方式发生了本质变化：

- 从“硬编码原型”
- 变成“可在Godot编辑器内持续制作地图、房间、遭遇、交互和成长内容的原型生产线”

对于一款系统密度较高的策略RPG来说，这种工具化推进，比单纯多做几张图、多写几段文本更重要。  
它决定了项目能否在有限时间里真正形成“完整可玩的Demo”，而不是停留在若干孤立功能点。

---

## 8. 可用于提案的简短总结版

本项目中，AI主要承担程序协作、工具链搭建、文档整理、系统调试与音频接入等工作。  
其中最关键的价值，不是替代创作者生成内容，而是帮助我快速将设计想法转化为Godot引擎内可编辑、可验证、可复用的设计工具。  
例如战斗房间作者工具、地图敌人生成图层、交互层、剧情触发区、图鉴与构筑界面等，均是在AI协作下快速完成原型并持续迭代。  
这种协作方式显著提升了独立开发中复杂系统的落地效率，使我能够把更多精力集中在玩法判断、题材表达与最终体验把控上。
