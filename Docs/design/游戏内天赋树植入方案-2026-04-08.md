# 游戏内天赋树植入方案

时间：2026-04-08

适用范围：
- 当前 demo 版本的局外成长系统
- 与现有 `C` 键综合菜单、卡组构筑、装备、学习机制对接
- 面向“真正植入到游戏内”的方案，而不是只停留在表格设计层

## 1. 文档目标

这份文档不再讨论“天赋树可以长什么样”，而是回答以下更实际的问题：

- 天赋树怎样才能真正嵌入到游戏内，而不是像网页原型
- 玩家在 demo 流程里什么时候看到它、为什么要点它
- 卡牌树、角色树、学习机制、装备、构筑之间怎样在游戏内形成一个闭环
- 现有实现基础上，哪些部分该保留，哪些部分该收敛

## 2. 当前判断

当前项目已经有一个“可操作的天赋树原型”，但它更像系统实验场，而不是完整的游戏内成长体验。

当前已有基础：

- `C` 键综合菜单已经存在
- 天赋节点、构筑、装备、图鉴已放在同一入口
- 角色成长已能影响攻击、防御减伤、构筑预算等基础数值
- 天赋树已经支持前置关系、节点购买和部分卡牌解锁

当前主要问题：

- 天赋树当前更像“测试工具”，而不是叙事和战斗循环中的自然组成部分
- 节点内容仍偏原型化，很多节点只是临时占位
- 卡牌树与角色树虽然在规则上互通，但在游戏内感知上还不够强
- 玩家很难快速理解“我为什么要现在点这个节点”

所以接下来不应继续把天赋树做得更大，而应把它做得更“像游戏内成长系统”。

## 3. 总体方案

推荐采用：

- `双树同屏`
- `共用专精点`
- `分阶段开放`
- `递归树状分支`
- `节点少而关键`

也就是说：

- 保留两棵树：
  - 卡牌树
  - 角色能力树
- 但不要做成两个分开的 panel
- 而是放在同一张可拖拽、可缩放的成长图上
- 节点关系不再是“中轴主干 + 一层侧枝”
- 而是允许：
  - 主干节点继续分叉
  - 分叉节点继续分叉
  - 多层递归生长
- 让玩家在一个空间里感受到“构筑成长”和“角色成长”其实属于同一套 build

## 4. 游戏内定位

天赋树在游戏内的职责不是“另一个菜单”，而是：

- 给玩家提供中长期 build 方向
- 决定高阶牌能否正常携带
- 决定角色基础动作和资源节奏
- 把探索、战斗奖励、学习机制和构筑真正串起来

它应该服务这条链：

`探索 -> 战斗 -> 奖励 -> 天赋分配 -> 构筑调整 -> 下一场战斗`

## 5. 入口方案

## 5.1 主要入口

当前最适合保留：

- `C` 键综合菜单

原因：

- 玩家已经接受这个入口
- 当前 demo 体量不适合再拆成多个独立子菜单入口
- 天赋、装备、构筑本来就应该互相贴近

## 5.2 天赋页在综合菜单中的位置

建议保留为一级标签页，但要强化“成长中枢”的地位：

- 角色状态
- 背包
- 构筑
- 天赋
- 图鉴

其中：

- `角色状态 / 构筑 / 天赋`
  这三页属于 build 核心
- `背包 / 图鉴`
  属于外围支撑

## 5.3 何时强制引导玩家打开

demo 中建议有三次明确引导：

1. 第一次遇到荒川后
- 用剧情或提示引导玩家第一次打开天赋页
- 只让玩家做一个非常轻量的选择

2. 第一次较完整的战斗奖励后
- 明确让玩家知道“这次升级点会改变你的打法”

3. Boss 前
- 给玩家最后一次完整 build 调整机会

## 6. 结构方案

## 6.1 双树同屏

推荐正式结构：

- 中间区域：卡牌树根节点
- 另一侧区域：角色能力树根节点
- 两边在视觉上彼此独立，但通过跨树节点或共用说明联结
- 每棵树内部都采用真正的树状蔓生结构
- 不要求所有节点严格落在一条主线上

不建议：

- 左边一个 panel，右边一个 panel，像配置工具
- 完全分离的两棵树页面

因为那样会把成长体验切碎。

## 6.2 根节点设计

每棵树都应该有真正的根节点样式，而不是一行标题文字。

推荐：

- `卡牌树`
  - 名称：`战术牌库`
  - 说明：决定你能学会并正常驾驭哪些牌
- `角色树`
  - 名称：`生存框架`
  - 说明：决定你如何承载这些牌并活着打出来

这样玩家一眼能理解两棵树的分工。

## 6.3 视觉布局

推荐：

- 同一个可拖拽画布
- 鼠标拖动画布移动
- 滚轮缩放
- 右侧固定详情 panel

这比传统列表式天赋页更像“成长地图”，也更符合现在已经做出的操作方式。

## 7. 节点内容设计

## 7.1 节点分类

建议所有节点只分四类：

1. `入口节点`
- 解锁一个方向的最小玩法语言

2. `专精节点`
- 提供分支专精值
- 给高阶牌正常携带资格开门

3. `关键节点`
- 解锁招牌牌、招牌动作或构筑权限

4. `角色节点`
- 提供生命、能量、移动、构筑预算、基础动作强化等

这样足够直观，不需要更多花哨分类。

## 7.2 树状生长规则

新版结构原则：

- 任何节点都可以长出后继节点
- 主节点可以分叉
- 分叉节点也可以继续分叉
- 节点的价值不由它是不是“主干”决定，而由：
  - 它是不是 build 入口
  - 它是不是专精门槛
  - 它是不是关键玩法节点
  - 它是不是高层收束节点

推荐把每一棵树理解成：

- 根节点
- 多条成长路径
- 若干路径在中后段汇合
- 或者保持平行直到高层由跨树节点重新联结

这样做的好处：

- 更像游戏内成长地图，而不是流程图
- 玩家每次点节点都更像“偏向某种打法”
- 更容易做出同一流派内的不同 build

## 7.3 卡牌树节点应承担的事

卡牌树节点最适合承担：

- 解锁新牌
- 开启正常携带资格
- 增加同类牌可携带数量
- 减少该流派牌的构筑压力
- 解锁学习相关能力

不建议卡牌树大量承担：

- 纯数值百分比增伤
- 与牌无关的泛属性成长

## 7.4 角色树节点应承担的事

角色树节点最适合承担：

- 最大生命
- 初始能量
- 移动力
- 普通攻击
- 防御动作减伤比例
- 冥想收益
- 构筑预算
- 超规携带槽位

这样角色树是“承载 build 的骨架”，而不是另一套牌树。

## 8. 与现有三分支的关系

现有三分支继续保留：

- 毁灭 / `melee`
- 制衡 / `ranged`
- 创造 / `flex`

## 8.1 卡牌树中三分支的职责

### 毁灭

负责：

- 贴身压迫
- 护盾压制
- 反击收益
- 撞击与终结

### 制衡

负责：

- 直线火力
- 距离控制
- 先手压制
- 标定与单点击杀

### 创造

负责：

- 地形改写
- 过牌与资源调度
- 学习机制
- 障碍物互动

## 8.2 角色树不再分三系主线

角色树建议不要再复制三条大分支。

应该采用：

- 共通主干
- 少量带流派门槛的分支节点

例如：

- `应激输出`
- `生还者框架`
- `稳态电容`
- `趋前步伐`
- `余量缓存`
- `老练本能`
- `荒川增幅`

## 9. 共用资源方案

## 9.1 成长货币

继续采用：

- `专精点`

专精点统一用于：

- 卡牌树节点
- 角色树节点
- 跨树关键节点

## 9.2 为什么必须共用

如果卡牌树和角色树各自独立给点数，玩家会觉得：

- “这边也点一点，那边也点一点”
- 没有真正取舍

共用资源的价值在于：

- 让每次升级都是 build 方向选择
- 让玩家感到“这次我更偏向构筑还是偏向角色”

## 10. 与构筑系统的正式关系

天赋树不只是解锁卡牌，更关键的是影响构筑合法性。

推荐正式关系：

- 卡牌树：
  - 决定某牌是否正常携带
  - 决定某流派牌的复制上限
  - 决定某流派牌的影响因子是否减免
- 角色树：
  - 决定整体构筑预算
  - 决定超规携带槽位
  - 决定某些基础动作是否能支撑高费构筑

也就是说：

- 天赋树不是“解锁卡牌库”
- 而是“决定你是否真正驾驭这套牌”

## 11. 与学习机制的关系

学习机制应挂在创造树里，但不应只是一张孤立牌。

推荐结构：

- 创造树入口：让玩家能接触资源调度和地形
- 创造树中层：解锁 `学习`
- 创造树深层：降低学习代价或提高学习收益
- 角色树侧：
  - 提供学习后的携带承载
  - 或减轻超规学习牌的惩罚

这样玩家才能感到：

- 学会敌方能力
- 只是第一步
- 后续还要靠成长树真正把它纳入 build

## 12. 节点数量控制方案

当前 demo 不适合做一整棵巨大天赋树。

推荐总量：

- 卡牌树：
  - 每条分支 `6 ~ 8` 个正式节点
  - 其中至少包含 2 层以上递归分叉
- 角色树：
  - `8 ~ 10` 个节点
- 跨树节点：
  - `1 ~ 2` 个展示型节点

总节点数控制在：

- `24 ~ 32`

这个数量已经足够形成真正的树状结构，同时仍然不会把 demo 做成不可读的大网。

## 13. 游戏内交互方案

## 13.1 默认状态

- 左键点节点：选中节点
- 不直接购买

## 13.2 详情面板

选中节点后，右侧详情 panel 显示：

- 名称
- 类型
- 前置
- 花费
- 解锁内容
- 数值收益
- 当前状态

## 13.3 解锁方式

必须保留二段确认：

- 第一步：点节点，只选中
- 第二步：按“解锁”按钮

这个设计当前已经是正确方向，应继续保留。

## 13.4 节点状态

节点只保留三种强视觉状态：

- `已解锁`
- `可购买`
- `锁定`

建议同时用：

- 颜色
- 亮度
- 描边粗细

三种方式共同表达，不只靠颜色。

## 14. demo 版推荐成长节奏

## 14.1 序章

只给玩家一次轻量成长体验：

- 让玩家在三系里选一个入口节点
- 不要求做深

目的：

- 让玩家知道这游戏不是纯战斗样机

## 14.2 星港探索

开放第一次完整构筑 + 天赋调整：

- 允许补第 2 层节点
- 允许做第一次明确 build 偏向

## 14.3 下水道精英后

开放能力点节点或角色树第 2 层：

- 玩家开始真正感到“这套牌是我的 build”

## 14.4 Boss 前

允许做一次最终 build 微调：

- 决定是保守成型
- 还是冒风险带学习牌 / 超规牌

## 15. 现有实现建议保留与重做部分

## 15.1 建议保留

- `C` 键综合菜单入口
- 同屏拖拽 / 缩放天赋树视图
- 详情 panel + 解锁按钮
- 天赋树节点与构筑系统之间的直接联动

## 15.2 建议重做

- 现有节点内容命名和层次应对齐设计 wiki
- 当前 `SystemFeatureLabController` 中的原型节点应逐步替换为正式节点集
- 不再继续扩“实验型节点”，而应改为围绕 demo 正式节奏整理

## 16. 落地顺序建议

如果要从当前实现继续推进，建议按这个顺序：

1. 先收敛节点表
- 把当前原型节点替换成正式节点集

2. 再收敛 UI 结构
- 双树根节点
- 详情 panel
- 三状态视觉

3. 再补节点效果
- 解锁牌
- 角色属性
- 构筑权限

4. 最后补跨树节点
- 跨树节点不该先做

## 17. 最终结论

真正适合植入到游戏内的天赋树，不应该像网页原型那样只是“点一堆技能图标”，而应该是：

- 有明确入口
- 有明确成长节奏
- 与构筑、装备、学习、战斗奖励共用同一条闭环

对当前项目最合适的正式方案是：

- 保留双树
- 保留共用专精点
- 保留同屏成长地图
- 严格控制节点数量
- 强化“卡牌树决定你会什么，角色树决定你扛不扛得住”

只要按这个方案收敛，天赋树就会从“测试系统”变成真正服务 demo 体验的成长中枢。
